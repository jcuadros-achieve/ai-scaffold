# Internationalization (i18n) rules

> Optional module — relevant to projects that serve more than one language or
> locale. Run `ai-init` to align with the project's i18n library.

Retrofitting i18n is expensive; designing for it is cheap. Even single-locale
projects benefit from not hardcoding user-facing text.

## Strings

- **No hardcoded user-facing strings.** Externalize to message catalogs keyed by
  id; the code references keys, not literals.
- **Never build sentences by concatenation.** Word order and grammar vary by
  language. Use full templated messages with named placeholders.
- Handle **pluralization** through the i18n library's plural rules, not
  `count === 1 ? 'item' : 'items'`.

## Locale-aware formatting

- Format dates, times, numbers, and currency through locale-aware APIs
  (`Intl.*` or the library) — never hand-rolled.
- Store and compute in canonical forms (UTC, minor currency units); localize
  only at display.

## Layout & content

- Allow text expansion (translations can be ~30% longer); don't hardcode widths
  that clip.
- Support RTL where the audience needs it (logical CSS properties, `dir`).
- Keep locale data and translations out of code, in versioned resource files.

> Run ai-init to record the i18n library, supported locales, and catalog layout.
